# Asset Generation Architecture

This document tracks the generation logic for various engineering applications within the platform. It outlines which parts of the codebase generate which physical deliverable assets, detailing the inputs, data sources, and dependencies.

---

## 1. Water Applications

### 1.1 Booster Set

The Booster Set configurator takes a set of parameters (number of pumps, motor power, motor start type, enclosure specs) and orchestrates the generation of an Asset Pack (a downloadable `.zip` file) containing specifications, parts lists, and diagrams. 

Currently, the React UI (`BoosterSetPage.jsx`) sends the complete configuration to the FastAPI backend without filtering by the checked assets in the UI. The backend generates all eligible documents and streams them back to the frontend, which merges them with client-side SVGs before downloading.

#### 1. The Core Engineering Logic: The "Digital Twin" DNA
Before any files are generated, the backend must mathematically resolve the physical requirements of the panel.
*   **Method**: `ConfigurationEngine.generate_twin()` inside `back/app/services/rule_resolver.py`.
*   **Input**: The user's form data (`series_id`, `motor_power_kw`, `load_count`/pumps, `ats_included`).
*   **Logic & Sheets Used**: 
    1.  Queries the mapped `StarterOption` database (from `Configurations.csv`) to find the exact base Starter (contactors, breakers) for the power rating. 
    2.  Checks `ConfigurationRule` (from `Enclosure_Rules.csv`) to determine the cabinet size that can fit those starters. 
    3.  Evaluates text-based math formulas in `AccessoryRule` (from `Accessory_Rules.csv`) to calculate quantities for items like pushbuttons or terminal blocks based on `load_count`.
*   **Output**: A massive, nested JSON object (`DigitalTwinResponse`) containing the complete architectural blueprint.

#### 2. The Excel Asset Pack (BOM & Data)
Fulfills the **"Data Sheet"** and **"Bill of Materials"** assets.
*   **Generated By**: `generate_excel_from_twin(twin)` in `back/app/generators/excel_gen.py`.
*   **Exported As**: `Neutral/ApplicationPack_{id}.xlsx` within the final ZIP.
*   **Logic**: Uses `pandas` to map the Digital Twin object into a 6-sheet Excel workbook in-memory.
    *   **Sheet 1 ("Parameters")**: Maps high-level metadata (Power, Starter Type, Enclosure dimensions).
    *   **Sheet 2 ("BOM-Template")**: Loops through the Digital Twin's `enclosure`, `components`, and `accessories` arrays, creating a unified parts list.
    *   **Sheet 3 ("IO-List")**: Detailed I/O mapping for the PLC/Controller.
    *   **Sheet 4 ("Network-IP-Plan")**: Maps the IO tagging from `ApplicationIOTemplate` to a network structure, assigning IP addresses and communication parameters based on the selected communication protocol.
    *   **Sheet 5 ("Alarm_List")**: Generates a project-specific alarm list using `ApplicationAlarmTemplate`, expanding templates for multi-pump configurations.
    *   **Sheet 6 ("Option-Matrix")**: Lists all standard and optional extras configured for the panel using `ApplicationOptionMatrix`.

#### 3. The Engineering Specification Word Document
Fulfills the **"Specification"** asset.
*   **Generated By**: `generate_word_from_twin(twin)` in `back/app/generators/word_gen.py`.
*   **Exported As**: `004_EngineeringSpec_{id}.docx` within the final ZIP.
*   **Logic**: Uses `python-docx` to read the Digital Twin JSON and programmatically write standard textual paragraphs regarding the enclosure size and panel ratings. It then loops through the Twin parts array to dynamically draw a Word BOM Table.

#### 4. BIM Object (IFC)
Fulfills the **"BIM Object"** asset.
*   **Generated By**: `generate_ifc_from_twin(twin)` in `back/app/generators/bim_gen.py`.
*   **Exported As**: `BIM/015_Logical_{id}.ifc` and `BIM/016_Visual_{id}.ifc` within the final ZIP.
*   **Logic**: Uses `ifcopenshell` to generate IFC4X3 compliant files.
    *   **Logical**: Focuses on Psets (Property Sets) for electrical data and port locations.
    *   **Visual**: Includes 3D mesh representations (boxes) for visibility in BIM coordination tools.

#### 5. JSON Payload & Manifest
*   **Generated By**: `ZipService.create_project_package()` in `back/app/services/zip_service.py`.
*   **Exported As**: `001_manifest.json` and `002_DigitalTwin_DNA_{id}.json` within the final ZIP.
*   **Logic**: The Manifest lists generation timestamps and included file paths. The Twin DNA is a raw dump of the JSON engine output for potential downstream REST imports into other technical architectures.

#### 6. Multi Line Diagram (SVG / PNG)
Fulfills the **"Multi Line Diagram"** asset.
*   **Generated By**: The React Frontend natively, specifically `handleDownload()` in `front/src/pages/BoosterSetPage.jsx`.
*   **Exported As**: `Docs/MultiLineDiagram.svg` and `Docs/MultiLineDiagram.png` within the final ZIP.
*   **Logic**: React draws an SVG graphic directly on the webpage based on configuration inputs. Upon downloading, `JSZip` intercepts the backend `.zip` stream, extracts the raw UI `<svg>` element, draws it to an invisible HTML5 canvas to yield a `.png`, and injects both natively into the final ZIP archive payload.

*(Note: "Drawings" is currently listed in the UI checklist, but is not generated.)*
